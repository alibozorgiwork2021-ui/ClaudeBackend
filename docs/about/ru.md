# ClaudeBackend — что это такое и зачем им пользоваться

[English](en.md) · [فارسی](fa.md) · [日本語](ja.md) · [中文](zh.md) · **Русский** · [Français](fr.md) · [Deutsch](de.md)

> Универсальная мультиагентная **система разработки бэкенда**: передайте ей
> репозиторий и сформулированную обычным языком задачу, и она реализует изменение
> в git-ветке, готовой к ревью — с учётом зависимостей, с проверкой и без
> единого касания вашего рабочего дерева.

## Что такое ClaudeBackend?

ClaudeBackend — это агент командной строки, который принимает кодовый репозиторий
и произвольную задачу, сформулированную обычным языком — «Добавь аутентификацию
по JWT», «Отрефактори модели SQLAlchemy», «Добавь эндпоинт `/health`» или даже
«Перенеси это с Python 2 на 3» — и реализует её. В его основе лежит большая
языковая модель (по умолчанию Claude Opus 4.8, с контекстным окном в 1 млн
токенов), обёрнутая в **детерминированный изолированный конвейер из трёх
агентов**: **Planner** решает, какие файлы создать, изменить или удалить;
**Coder** реализует каждый шаг; а **Verifier** в роли страховочной сетки
запускает проверки синтаксиса, `ruff` и собственный набор тестов `pytest` проекта
(с числом повторов до 3). Модель выполняет собственно написание кода; окружающая
её программа решает, *что* менять, *в каком порядке*, и *проверяет результат*.
Результат записывается в **новую git-ветку** — ваше рабочее дерево и текущая ветка
остаются нетронутыми.

## Какую проблему она решает

Реальная работа над бэкендом редко умещается в один файл. Добавление эндпоинта,
смена схемы аутентификации, перекройка модели данных или модернизация
унаследованной кодовой базы — всё это расходится волнами по модулям, ORM-моделям,
конфигурации и тестам. Делать это вручную медленно и чревато ошибками; доверить
это наивному ассистенту кода рискованно, ведь ассистент правит по одному файлу за
раз и не видит, как изменение в одном месте ломает другое.

Опасные ошибки — это те, что пересекают границы файлов:

> Вспомогательная функция возвращает `d.keys()`. В Python 2 это `list`, поэтому
> другой модуль спокойно обращается к `keys()[0]`. В Python 3 `keys()` — это
> *представление* (view): `keys()[0]` вызывает `TypeError`. Чисто локальный
> инструмент «чинит» оба файла и оставляет кодовую базу сломанной, потому что
> ошибка проявляется только тогда, когда оба файла рассматриваются *вместе*. Та
> же ловушка таится в бесчисленных изменениях бэкенда — переименуйте поле модели,
> и каждый запрос и сериализатор, что к нему обращался, может молча сломаться.

## Чем ClaudeBackend отличается

| | наивные ассистенты кода | линтеры (например, SonarQube) | ClaudeBackend |
|---|---|---|---|
| Реализует изменение (а не просто правит/отчитывается) | по одному файлу за раз | только чтение | сквозь весь репозиторий, от начала до конца |
| Межфайловые / учитывающие зависимости исправления | нет | нет | да — карта импортов, ORM, конфигов |
| Помечает неоднозначные / рискованные решения | нет | нет | да (`CLAUDEBACKEND-REVIEW`) |
| Результат | правки на месте | отчёт | git-ветка для ревью + сводка |

Ключевая идея: ClaudeBackend строит **граф зависимостей** вашего кода — он
отображает импорты Python *и* ORM-модели (Django / SQLAlchemy), Dockerfile и
конфигурационные файлы — и передаёт этот реальный контекст Planner'у. Каждый файл
показывается модели *вместе с его зависимостями* внутри очень большого
контекстного окна. Именно поэтому он способен реализовывать изменения, которые
расходятся по файлам, вместо сломанных правок по одному файлу, которые выдают
чисто локальные инструменты.

## Кому это нужно

- **Командам, выпускающим бэкенд-функциональность**, которым нужна ветка для ревью,
  а не «чёрный ящик» массовых правок.
- **Сопровождающим**, которые модернизируют сервисы, перекраивают модели данных
  или выплачивают технический долг сразу во многих файлах.
- **Консультантам и подрядчикам**, выполняющим крупные рефакторинги или миграции и
  желающим получить diff, готовый к ревью, а не «чёрный ящик».
- **Всем**, у кого есть унаследованная кодовая база — включая утилиту на Python 2,
  которая «всё ещё работает», но больше не устанавливается на современной машине —
  и кому нужно аккуратное обновление с учётом зависимостей.

## Ключевые возможности

- **Разработка с учётом зависимостей и межфайловыми правками** — главная
  особенность: она отображает импорты, ORM-модели, Dockerfile и конфиги, чтобы
  Planner видел реальный контекст.
- **Конвейер из трёх агентов** — Planner, Coder и Verifier работают как
  изолированные детерминированные этапы, так что любая задача проходит один и тот
  же выверенный путь.
- **Честная, многоуровневая верификация** — синтаксический барьер для каждого
  файла, затем сквозная проверка всего проекта (компиляция + `ruff` + ваш
  собственный набор тестов `pytest`, если он собирается), с числом повторов до 3
  в роли страховочной сетки.
- **Безопасность по построению** — отказывается работать при «грязном» рабочем
  дереве, пишет только в новую ветку (`claudebackend/feature-<timestamp>`) и имеет
  режим `--dry-run` (по умолчанию для агентов), который ничего не записывает.
- **Помечает то, в чём не уверен** — неоднозначные или связанные с безопасностью
  изменения реализуются *и* помечаются комментарием `CLAUDEBACKEND-REVIEW` для
  подтверждения человеком.
- **Используйте собственную LLM** — Claude по умолчанию; а также другие
  совместимые с OpenAI провайдеры (OpenRouter, OpenAI, NVIDIA, DeepSeek и Gemini).
- **Используйте из своих инструментов** — поставляется как MCP server, как Agent
  Skill и как Claude Code plugin, так что Cursor, Codex, Google Antigravity и
  Claude Code/Desktop могут его вызывать.

## Как это работает (вкратце)

1. **Граф** — построение карты зависимостей репозитория: импорты Python (с помощью
   стандартного `tokenize`, чтобы можно было разобрать даже исходники Python 2,
   которые `ast` отвергает), ORM-модели (Django / SQLAlchemy), Dockerfile и
   конфигурационные файлы. Циклы импортов сворачиваются в единый блок.
2. **План** — Planner превращает вашу задачу в конкретный список файлов, которые
   нужно создать, изменить или удалить, с пометками о риске и заметками по каждому
   файлу.
3. **Разработка** — для каждого шага Coder собирает контекст (сам файл плюс его
   зависимости, с кэшированием промптов), потоково генерирует изменение, проверяет
   его синтаксис и при сбое выполняет повторную попытку.
4. **Верификация** — сквозная компиляция + линтинг + прогон тестов по всему
   проекту: настоящий межфайловый барьер (с числом повторов до 3).
5. **Коммит** — создание ветки, коммит по каждому модулю и запись `DEV_SUMMARY.md`,
   а также интерактивного графа топологии `DEV_GRAPH.md`.

## Честно о своих ограничениях

Статические проверки — это **страховочная сетка, а не доказательство
корректности**. Проверки синтаксиса и `ruff` отлавливают определённый класс
ошибок, но сохраняющие поведение, но при этом неоднозначные решения принимаются
моделью и *помечаются* для вас, а не доказываются как корректные. Самая надёжная
гарантия — это **прохождение вашего собственного набора тестов** после изменения.
ClaudeBackend создан для того, чтобы сделать это ревью быстрым и честным, а не для
того, чтобы делать вид, будто работа над бэкендом полностью автоматизируема.

## С чего начать

```bash
# 1. Установка (загрузочный скрипт под каждую ОС — см. руководства по установке):
#    Windows: powershell -ExecutionPolicy Bypass -File scripts\setup.ps1
#    macOS:   ./scripts/setup-macos.sh
#    Linux:   ./scripts/setup-linux.sh

# 2. Аутентификация (например, с помощью ключа Anthropic API) и предпросмотр работы:
export ANTHROPIC_API_KEY=...
claudebackend develop path/to/repo "Add a /health endpoint" --dry-run  # ничего не записывает
```
