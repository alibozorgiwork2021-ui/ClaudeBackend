# ClaudeBackend — その正体と、使うべき理由

[English](en.md) · [فارسی](fa.md) · **日本語** · [中文](zh.md) · [Русский](ru.md) · [Français](fr.md) · [Deutsch](de.md)

> 汎用的なマルチエージェント型**バックエンド開発システム**。リポジトリと自然言語で
> 書いた目的を渡せば、その変更をレビュー可能な git ブランチ上に実装します。依存関係を
> 把握し、検証済みで、あなたの作業ツリーには一切触れません。

## ClaudeBackend とは何か

ClaudeBackend は、コードリポジトリと、自然言語で書かれた任意の目的を受け取るコマンド
ライン エージェントです。たとえば「JWT 認証を追加して」「SQLAlchemy のモデルを
リファクタリングして」「`/health` エンドポイントを追加して」、さらには「これを
Python 2 から 3 へ移行して」といった目的を、そのまま実装します。大規模言語モデル
（デフォルトでは Claude Opus 4.8、1M トークンのコンテキストウィンドウ）を、
**決定論的で隔離された 3 エージェントのパイプライン**で包み込んでいます。すなわち、
**Planner** がどのファイルを作成・変更・削除するかを決め、**Coder** が各ステップを
実装し、**Verifier** がセーフティネットとして構文チェック・`ruff`・プロジェクト自身の
`pytest` スイートを実行します（最大 3 回までリトライ）。実際のコーディングはモデルが
担い、それを取り囲むプログラムが、*何を*変更し、*どの順序で*行い、*結果を検証する*かを
決定します。出力は**新しい git ブランチ**へ書き込まれ、あなたの作業ツリーや現在の
ブランチには一切手が加えられません。

## 解決する課題

実際のバックエンド作業が 1 ファイルで完結することはまれです。エンドポイントの追加、
認証方式の差し替え、データモデルの作り直し、レガシーコードベースの近代化——どれもが
モジュール、ORM モデル、設定、テストへと波及します。これを手作業で行うのは遅く、
誤りも生じやすい。素朴なコードアシスタントに任せるのもリスクがあります。なぜなら
アシスタントは一度に 1 ファイルしか編集できず、ある場所の変更がどのように別の場所を
壊すかを見通せないからです。

危険なバグとは、ファイルの境界をまたぐものです。

> あるヘルパー関数が `d.keys()` を返すとします。Python 2 ではこれは `list` なので、
> 別のモジュールは安全に `keys()[0]` と書けます。Python 3 では `keys()` は*ビュー*で
> あり、`keys()[0]` は `TypeError` を送出します。純粋に局所的なツールは両方のファイルを
> 「修正」しつつ、コードベースを壊れたまま放置します。なぜなら、このバグは 2 つの
> ファイルを*一緒に*見たときにしか現れないからです。同じ罠は数えきれないほどの
> バックエンド変更に潜んでいます——モデルのフィールド名を変えれば、それに触れていた
> すべてのクエリやシリアライザが静かに壊れかねません。

## ClaudeBackend が違う理由

| | 素朴なコードアシスタント | リンター（例: SonarQube） | ClaudeBackend |
|---|---|---|---|
| 変更を実装する（編集・報告だけではない） | 一度に 1 ファイル | 読み取り専用 | リポジトリ全体をまたいで一貫して |
| ファイルをまたぐ／依存関係を考慮した修正 | 不可 | 不可 | 可能 — import・ORM・設定をマッピング |
| 曖昧／リスクのある選択にフラグを立てる | 不可 | 不可 | 可能（`CLAUDEBACKEND-REVIEW`） |
| 出力 | その場で編集 | レポート | レビュー可能な git ブランチ＋サマリー |

中核となる考え方はこうです。ClaudeBackend はあなたのコードの**依存グラフ**を構築します
——Python の import *だけでなく* ORM モデル（Django / SQLAlchemy）、Dockerfile、設定
ファイルまでをマッピングし——その実際のコンテキストを Planner に与えます。各ファイルは、
*その依存ファイルと一緒に*、非常に大きなコンテキストウィンドウの中でモデルに見せられます。
だからこそ、純粋に局所的なツールが生み出す、ファイルごとの壊れた編集ではなく、ファイルを
またいで波及する変更を実装できるのです。

## 誰に必要か

- ブラックボックスの一括編集ではなく、レビュー可能なブランチを求める、**バックエンド
  機能を出荷するチーム**。
- サービスを近代化したり、データモデルを作り直したり、多数のファイルにまたがる技術的
  負債を返済したりする**メンテナー**。
- 大規模なリファクタリングや移行を手がけ、ブラックボックスではなくレビュー可能な差分を
  求める**コンサルタントや受託開発者**。
- レガシーなコードベースを抱える**すべての人**——「まだ動く」が最新のマシンではもう
  インストールできない Python 2 ユーティリティを含め、慎重で依存関係を考慮した更新を
  必要とする人。

## 主な機能

- **依存関係を考慮した、ファイルをまたぐ開発** — 目玉となる機能です。import、ORM モデル、
  Dockerfile、設定をマッピングし、Planner が実際のコンテキストを見られるようにします。
- **3 エージェントのパイプライン** — Planner、Coder、Verifier が隔離された決定論的な
  ステージとして動くため、あらゆる目的が同じ規律ある道筋をたどります。
- **誠実で多層的な検証** — ファイルごとの構文ゲート、続いてプロジェクト全体のパス
  （コンパイル＋`ruff`＋収集できる場合はあなた自身の `pytest` スイート）。セーフティ
  ネットとして最大 3 回までリトライします。
- **構造的に安全** — 汚れた作業ツリーを拒否し、新しいブランチ
  （`claudebackend/feature-<timestamp>`）にのみ書き込み、何も書き込まない `--dry-run`
  モード（エージェントにとってのデフォルト）を備えています。
- **確信が持てない点にフラグを立てる** — 曖昧な、あるいはセキュリティに関わる変更は
  実装*した上で*、人間が確認するための `CLAUDEBACKEND-REVIEW` コメントで印を付けます。
- **自分の LLM を使える** — デフォルトは Claude ですが、他の OpenAI 互換プロバイダー
  （OpenRouter、OpenAI、NVIDIA、DeepSeek、Gemini）も使えます。
- **自分のツールから呼び出せる** — MCP server、Agent Skill、Claude Code plugin として
  提供されるため、Cursor、Codex、Google Antigravity、Claude Code/Desktop から呼び出せます。

## 仕組み（概観）

1. **グラフ化** — リポジトリの依存関係をマッピングします。Python の import（標準
   ライブラリの `tokenize` を使うため、`ast` が拒否する Python 2 のソースさえ解析
   できます）、ORM モデル（Django / SQLAlchemy）、Dockerfile、設定ファイルまで。
   import の循環は単一のユニットへとまとめられます。
2. **計画** — Planner があなたの目的を、作成・変更・削除すべきファイルの具体的な
   リストへと落とし込み、ファイルごとのリスクとメモを注釈付けします。
3. **開発** — 各ステップについて、Coder がコンテキスト（そのファイルと依存ファイル、
   プロンプトキャッシュ付き）を構築し、変更をストリーミングし、構文チェックし、失敗時には
   リトライします。
4. **検証** — プロジェクト全体のコンパイル＋リント＋テストのパス。これが本物の、
   ファイルをまたぐゲートです（最大 3 回までリトライ）。
5. **コミット** — ブランチを作成し、モジュールごとにコミットし、`DEV_SUMMARY.md` と
   対話的なトポロジーグラフ `DEV_GRAPH.md` を書き込みます。

## その限界について誠実であること

静的チェックは**安全網であって、正しさの証明ではありません**。構文チェックと `ruff` は
一定の種類のミスを捕捉します。しかし、振る舞いを保ちつつも曖昧な選択は、モデルが判断し、
あなたのために*フラグを立てる*のであって、正しいと証明されるわけではありません。最も
確かな保証は、変更後にあなた**自身のテストスイートが通ること**です。ClaudeBackend は、
バックエンド作業を完全に自動化できると装うためではなく、そのレビューを素早く誠実なものに
するために作られています。

## はじめに

```bash
# 1. インストール（OS ごとのブートストラップスクリプト — インストールガイドを参照）:
#    Windows: powershell -ExecutionPolicy Bypass -File scripts\setup.ps1
#    macOS:   ./scripts/setup-macos.sh
#    Linux:   ./scripts/setup-linux.sh

# 2. 認証（例: Anthropic API キーを使用）して、まず作業をプレビューする:
export ANTHROPIC_API_KEY=...
claudebackend develop path/to/repo "Add a /health endpoint" --dry-run  # 何も書き込まない
```
