# ClaudeBackend — چیست و چرا از آن استفاده کنید

[English](en.md) · **فارسی** · [日本語](ja.md) · [中文](zh.md) · [Русский](ru.md) · [Français](fr.md) · [Deutsch](de.md)

> یک **سامانه‌ی توسعه‌ی بک‌اند** جامع و چندعاملی: یک مخزن کد و یک هدف به زبان ساده
> به آن بدهید، و آن تغییر را روی یک شاخه‌ی git قابل‌بازبینی پیاده‌سازی می‌کند — آگاه از
> وابستگی‌ها، راستی‌آزمایی‌شده، و بدون آنکه هرگز درخت کاری شما را دست بزند.

## ClaudeBackend چیست؟

ClaudeBackend یک عامل خط فرمان است که یک مخزن کد به‌همراه یک هدفِ دلخواه که به زبان
ساده نوشته شده را می‌گیرد — «احراز هویت JWT را اضافه کن»، «مدل‌های SQLAlchemy را
بازآرایی (refactor) کن»، «یک endpoint با مسیر `/health` اضافه کن»، یا حتی «این را از
Python 2 به 3 مهاجرت بده» — و آن را پیاده‌سازی می‌کند. این ابزار با یک مدل زبانی بزرگ
کار می‌کند (به‌طور پیش‌فرض Claude Opus 4.8، با پنجره‌ی زمینه‌ی یک‌میلیون توکنی) که درون یک
**خط لوله‌ی سه‌عاملیِ قطعی (deterministic) و ایزوله** پیچیده شده است: یک **Planner** تصمیم
می‌گیرد که کدام فایل‌ها ساخته، تغییر داده یا حذف شوند؛ یک **Coder** هر گام را پیاده‌سازی
می‌کند؛ و یک **Verifier** به‌عنوان یک تور ایمنی بررسی‌های نحوی، `ruff` و مجموعه‌ی `pytest`
خودِ پروژه را اجرا می‌کند (با حداکثر ۳ تلاش مجدد). مدل، کدنویسی واقعی را انجام می‌دهد؛
برنامه‌ی پیرامون آن تصمیم می‌گیرد که *چه چیزی* را تغییر دهد، *با چه ترتیبی*، و *نتیجه را
بررسی می‌کند*. خروجی در یک **شاخه‌ی git جدید** نوشته می‌شود — درخت کاری و شاخه‌ی فعلی شما
هرگز دست‌نخورده باقی می‌مانند.

## مشکلی که حل می‌کند

کار واقعیِ بک‌اند به‌ندرت در یک فایل جا می‌گیرد. افزودن یک endpoint، تعویض یک طرح احراز
هویت، بازشکل‌دهی یک مدل داده، یا نوسازی یک پایگاه‌کد قدیمی، همگی در سراسر ماژول‌ها، مدل‌های
ORM، پیکربندی و تست‌ها موج‌دار می‌شوند. انجام دستی آن کند و خطاپذیر است؛ انجام آن با یک
دستیار کد ساده‌انگارانه پرخطر است، زیرا آن دستیار هر بار یک فایل را ویرایش می‌کند و
نمی‌تواند ببیند که چگونه تغییری در یک نقطه، جای دیگری را می‌شکند.

باگ‌های خطرناک همان‌هایی هستند که از مرز فایل‌ها عبور می‌کنند:

> یک تابع کمکی `d.keys()` را برمی‌گرداند. در Python 2 این یک `list` است، پس ماژول دیگری
> به‌سادگی `keys()[0]` را می‌نویسد. در Python 3، `keys()` یک *view* است — `keys()[0]`
> خطای `TypeError` ایجاد می‌کند. یک ابزار صرفاً محلی هر دو فایل را «اصلاح می‌کند» و
> پایگاه‌کد را خراب باقی می‌گذارد، زیرا این باگ تنها زمانی نمایان می‌شود که به دو فایل
> *در کنار هم* نگاه کنید. همین دام در شمار بی‌پایانی از تغییرهای بک‌اند پنهان است — یک
> فیلد مدل را تغییر نام دهید، و هر کوئری و سریالایزری که با آن سروکار داشته می‌تواند
> بی‌سروصدا بشکند.

## چرا ClaudeBackend متفاوت است

| | دستیارهای کد ساده‌انگار | لینترها (مثلاً SonarQube) | ClaudeBackend |
|---|---|---|---|
| پیاده‌سازی یک تغییر (نه صرفاً ویرایش/گزارش) | هر بار یک فایل | فقط‌خواندنی | سرتاسری در کل مخزن |
| اصلاح‌های میان‌فایلی / آگاه از وابستگی | خیر | خیر | بله — نگاشت import، ORM، پیکربندی |
| علامت‌گذاری انتخاب‌های مبهم / پرخطر | خیر | خیر | بله (`CLAUDEBACKEND-REVIEW`) |
| خروجی | ویرایش در محل | یک گزارش | یک شاخه‌ی git قابل‌بازبینی + خلاصه |

ایده‌ی محوری: ClaudeBackend یک **گراف وابستگی** از کد شما می‌سازد — import‌های Python *و*
مدل‌های ORM (Django / SQLAlchemy)، فایل‌های Dockerfile و فایل‌های پیکربندی را نگاشت می‌کند —
و آن زمینه‌ی واقعی را در اختیار Planner می‌گذارد. هر فایل *در کنار وابستگی‌هایش* درون یک
پنجره‌ی زمینه‌ی بسیار بزرگ به مدل نشان داده می‌شود. به همین دلیل است که می‌تواند تغییرهایی
را پیاده‌سازی کند که در سراسر فایل‌ها موج‌دار می‌شوند، به‌جای ویرایش‌های خرابِ فایل‌به‌فایل که
ابزارهای صرفاً محلی تولید می‌کنند.

## چه کسانی به آن نیاز دارند

- **تیم‌هایی که قابلیت‌های بک‌اند را عرضه می‌کنند** و یک شاخه‌ی قابل‌بازبینی می‌خواهند، نه یک
  ویرایش انبوهِ جعبه‌سیاه.
- **نگهدارندگانی** که سرویس‌ها را نوسازی می‌کنند، مدل‌های داده را بازشکل می‌دهند، یا بدهی فنی
  را در سراسر فایل‌های متعدد می‌پردازند.
- **مشاوران و پیمانکارانی** که بازآرایی‌ها یا مهاجرت‌های بزرگ انجام می‌دهند و یک diff
  قابل‌بازبینی می‌خواهند، نه یک جعبه‌ی سیاه.
- **هر کسی** که یک پایگاه‌کد قدیمی دارد — از جمله یک ابزار کمکی Python 2 که «هنوز کار
  می‌کند» اما دیگر روی یک دستگاه مدرن نصب نمی‌شود — و به یک به‌روزرسانی دقیق و آگاه از
  وابستگی نیاز دارد.

## ویژگی‌های کلیدی

- **توسعه‌ی میان‌فایلی و آگاه از وابستگی** — قابلیت سرتیتر: import‌ها، مدل‌های ORM، فایل‌های
  Dockerfile و پیکربندی را نگاشت می‌کند تا Planner زمینه‌ی واقعی را ببیند.
- **خط لوله‌ی سه‌عاملی** — Planner، Coder و Verifier به‌صورت مراحل ایزوله و قطعی اجرا
  می‌شوند تا هر هدف از همان مسیر منظم پیروی کند.
- **راستی‌آزمایی صادقانه و لایه‌ای** — یک دروازه‌ی نحوی برای هر فایل، سپس یک گذر سراسری در
  سطح پروژه (کامپایل + `ruff` + مجموعه‌ی `pytest` خودتان، اگر جمع‌آوری شود)، با حداکثر ۳
  تلاش مجدد به‌عنوان تور ایمنی.
- **ایمن از پایه** — از یک درخت کاری ناتمیز سر باز می‌زند، فقط در یک شاخه‌ی جدید
  (`claudebackend/feature-<timestamp>`) می‌نویسد، و یک حالت `--dry-run` دارد (پیش‌فرض برای
  عامل‌ها) که هیچ‌چیز نمی‌نویسد.
- **علامت‌گذاری مواردی که از آن‌ها مطمئن نیست** — تغییرهای مبهم یا حساس از نظر امنیتی
  پیاده‌سازی می‌شوند *و* با یک کامنت `CLAUDEBACKEND-REVIEW` علامت می‌خورند تا یک انسان آن‌ها
  را تأیید کند.
- **از LLM خودتان استفاده کنید** — به‌طور پیش‌فرض Claude؛ همچنین دیگر ارائه‌دهندگان سازگار با
  OpenAI (OpenRouter، OpenAI، NVIDIA، DeepSeek و Gemini).
- **از درون ابزارهای خودتان به‌کارش ببرید** — به‌صورت یک MCP server، یک Agent Skill، و یک
  Claude Code plugin عرضه می‌شود، تا Cursor، Codex، Google Antigravity، و Claude
  Code/Desktop بتوانند آن را فراخوانی کنند.

## چگونه کار می‌کند (در یک نگاه)

1. **گراف** — نگاشت وابستگی‌های مخزن: import‌های Python (با `tokenize` کتابخانه‌ی استاندارد،
   تا حتی کد منبع Python 2 را که `ast` آن را رد می‌کند نیز بتواند تجزیه کند)، مدل‌های ORM
   (Django / SQLAlchemy)، فایل‌های Dockerfile و فایل‌های پیکربندی. چرخه‌های import در یک
   واحد واحد ادغام می‌شوند.
2. **برنامه‌ریزی** — Planner هدف شما را به یک فهرست مشخص از فایل‌هایی که باید ساخته، تغییر
   داده یا حذف شوند تبدیل می‌کند، که با ریسک و یادداشت‌های هر فایل حاشیه‌نویسی شده است.
3. **توسعه** — برای هر گام، Coder زمینه را می‌سازد (فایل به‌همراه وابستگی‌هایش، با کش کردن
   prompt)، تغییر را جریان‌دهی (stream) می‌کند، آن را بررسی نحوی می‌کند، و در صورت شکست
   تلاش مجدد می‌کند.
4. **راستی‌آزمایی** — یک گذر سراسری کامپایل + لینت + تست در سطح پروژه: دروازه‌ی واقعی
   میان‌فایلی (با حداکثر ۳ تلاش مجدد).
5. **commit** — ساخت شاخه، commit برای هر ماژول، و نوشتن یک `DEV_SUMMARY.md` به‌همراه یک
   گراف توپولوژی تعاملی `DEV_GRAPH.md`.

## صادقانه درباره‌ی محدودیت‌هایش

بررسی‌های ایستا یک **تور ایمنی هستند، نه اثباتِ درستی**. بررسی‌های نحوی و `ruff` دسته‌ای از
اشتباهات را می‌گیرند — اما انتخاب‌های حافظ‌رفتار اما مبهم توسط مدل تصمیم‌گیری می‌شوند و برای
شما *علامت* می‌خورند، نه آنکه درستی‌شان اثبات شود. مطمئن‌ترین تضمین، **قبول‌شدن مجموعه‌ی تست
خودتان** پس از تغییر است. ClaudeBackend برای آن ساخته شده که این بازبینی را سریع و صادقانه
کند، نه آنکه وانمود کند کار بک‌اند کاملاً خودکارشدنی است.

## شروع به کار

```bash
# 1. نصب (اسکریپت bootstrap برای هر سیستم‌عامل — راهنماهای نصب را ببینید):
#    Windows: powershell -ExecutionPolicy Bypass -File scripts\setup.ps1
#    macOS:   ./scripts/setup-macos.sh
#    Linux:   ./scripts/setup-linux.sh

# 2. احراز هویت (مثلاً با یک کلید API از Anthropic) و ابتدا پیش‌نمایش کار:
export ANTHROPIC_API_KEY=...
claudebackend develop path/to/repo "Add a /health endpoint" --dry-run  # هیچ‌چیز نمی‌نویسد
```
